# OWS Programmable Policy

> OWS Hackathon 2026-04-03

Extends the Open Wallet Standard's executable policy mechanism to let developers write custom signing policies in TypeScript. Evaluates AI agent signing requests in real time using external APIs, on-chain data, and dynamic policy chaining.

**AI agents get hacked. Prompt injection hijacks them. But OWS Programmable Policy doesn't use an LLM — it can't be tricked. It runs in an independent process, sees no prompts, no context. That's Zero Trust for Agents.**

## Challenges Addressed

| Challenge | How |
|-----------|-----|
| **#04 Agent identity attestation** | Live ERC-8004 Identity Registry verification on Base Sepolia — unregistered agents are blocked |
| **#01 Open Trust Standard** | Policy Chaining uses ERC-8004 reputation to dynamically adjust rules — same tx, different outcome based on trust |
| **#08 On-chain audit log** | Append-only SQLite audit log with tamper-proof triggers; extensible to on-chain Merkle Root |
| **#02 Agent intent verification** | Every signing request passes through KYC/AML/identity checks before the private key is ever decrypted |

## Architecture

```
Agent → OWS (ows sign) → Policy Engine → executable (this project)
                                              │
                                              ├── KYC Check (mock)
                                              ├── AML Check (mock)
                                              ├── ERC-8004 Agent ID (on-chain)
                                              └── Policy Chaining (dynamic rules)
                                              │
                                              └── Audit Log (SQLite, append-only)
```

ポリシー評価は **署名の直前・秘密鍵の復号前** に実行される。Deny されたらキーに一切触れずに拒否。エージェントはこのフローをバイパスできない。

## Policies

| Policy | Type | Description |
|--------|------|-------------|
| **KYC Check** | Mock | 送金先が KYC 済みかを外部 API/DB に照会 |
| **AML Check** | Mock | 送金先が制裁リストに該当しないかチェック |
| **ERC-8004 Agent ID** | On-chain | エージェントの ERC-8004 Identity Registry 登録・レピュテーション検証 |
| **Policy Chaining** | Dynamic | 前段ポリシーの結果に基づいてルールを動的変更 |

### Policy Chaining の動作

```
ERC-8004 reputation ≥ 80 → 高額送金も許可
ERC-8004 reputation 50-79 → 低額のみ許可、高額(>1 ETH)はブロック
ERC-8004 reputation < 50 → 全送金ブロック（ERC-8004 ポリシーで先に弾かれる）
```

同じトランザクションでも、エージェントのレピュテーションで結果が変わる。ビルトインの静的ルールでは不可能。

## Security: Zero Trust for Agents

| Threat | Mitigation |
|--------|-----------|
| Prompt injection | Policy runs in OWS process, not agent's. Agent cannot bypass. |
| PolicyContext tampering | Context is generated by OWS core, not the agent. |
| Policy config tampering | Config files are outside agent's API key scope. |
| Timeout exploit | Deny-by-default: timeout (5s) or error → always Deny. |
| Audit log tampering | SQLite triggers prohibit DELETE/UPDATE (append-only). |

## Setup

```bash
# Install OWS CLI
npm install -g @open-wallet-standard/core

# Install dependencies
npm install

npm run build

npm test
```

## Register with OWS

```bash
# Register the programmable policy
ows policy create --file ows-policy.json

# Create an API key with the policy attached
ows key create --name "my-agent" --wallet <wallet-name> --policy programmable-policy
```

## Future: OWS Signing Hooks Framework

What we built is effectively a **pre-sign hook**. We propose generalizing this into a full hooks framework:

| Hook | Status | Use Cases |
|------|--------|-----------|
| **pre-sign** | ✅ Implemented | KYC/AML, ERC-8004, Policy Chaining, approval flows |
| **post-sign** | Proposed | On-chain Merkle Root audit, notifications, analytics |
| **on-deny** | Proposed | Alert firing, agent suspension, incident response |

## Future Extensions

- **Human-in-the-Loop Approval**: High-risk transactions trigger a `pending` state, send approval requests via Slack/email/webhook, and wait for human authorization with signed, time-limited approval tokens. Proposes extending OWS `PolicyResult` with `{ action: "pending", callback_url }` for async approval flows.
- **On-chain Merkle Root**: Periodically write Merkle Root of audit log entries on-chain. Any log entry becomes provable via Merkle Proof — tamper-evident audit trail at minimal cost (1 tx/day).
- **Real KYC/AML APIs**: Replace mock with Chainalysis, Elliptic, Circle, Sumsub, or your own compliance API.
- **Policy Ecosystem**: Developers publish custom policies as npm packages. Install and register with one command.
