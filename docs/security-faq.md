# OWS OWS Hooks — Security FAQ & Attack Scenarios

## Attack Scenario Verification Results

Confirmed that deny-by-default functions correctly across all attack scenarios.

### 1. Malformed Input

| Attack | Input | Result |
|--------|-------|--------|
| Broken JSON | `THIS IS NOT JSON` | `deny: "Invalid JSON input"` |
| Empty input | `(empty)` | `deny: "Empty input"` |
| Missing required fields | `{"chain_id":"eip155:1"}` | `deny: "Missing required fields"` |
| Invalid value | `"value": "NOT_A_NUMBER"` | `deny: Policy threw (BigInt conversion error)` |

### 2. Injection

| Attack | Input | Result |
|--------|-------|--------|
| SQL injection | `api_key_id: "agent; DROP TABLE audit_log;--"` | Recorded as a string in the log. Table intact. Prevented by better-sqlite3 parameter binding. |
| Prototype pollution | `"__proto__": {"isAdmin": true}` | Ineffective because JSON.parse returns a plain object. No impact on policy logic. |

### 3. Audit Log Tampering

| Attack | Result |
|--------|--------|
| `DELETE FROM audit_log` | `ABORT: audit_log is append-only: DELETE is prohibited` |
| `UPDATE audit_log SET result = 'allow'` | `ABORT: audit_log is append-only: UPDATE is prohibited` |

DELETE/UPDATE are completely blocked by SQLite triggers.

### 4. DoS

| Attack | Result |
|--------|--------|
| 200KB payload | Processed, but capped by OWS's 5-second timeout |

OWS itself times out after 5 seconds → automatic Deny. No additional size limit is needed on the OWS Hooks side (controlled by OWS).

## FAQ

### Q: Can prompt injection be used to bypass the policy?

**A: No.** The OWS Hooks does not use an LLM at all. Pure Node.js code receives JSON input and deterministically returns Approve/Deny. Natural language instructions (e.g., "ignore the policy") simply fail JSON parsing and result in a Deny.

More importantly: **The OWS Hooks acts as the last line of defense when an agent is hijacked via prompt injection.**

```
Attacker → prompt injection → AI agent (hijacked)
  → agent attempts to send funds to a sanctioned address
  → OWS sign → OWS Hooks → AML check → Deny ✅
```

No matter how badly the agent's judgment is manipulated, policy evaluation runs in an independent process and is completely unaffected by the agent's prompt or context. This is the core of "Zero Trust for Agents."

### Q: Can an agent bypass the policy and sign directly?

**A: No.** Signing is performed inside OWS's Policy Engine. Agents can only request signatures via the `ows sign` API, and as long as a policy is bound to an API key, policy evaluation will always be executed. The private key is only decrypted after policy evaluation.

### Q: Can an agent learn the contents of the policy?

**A: The policy's decision logic is not exposed.** All an agent receives is `{ allow: true/false, reason: "..." }`. It has no knowledge of which policies exist, what thresholds are set, or what criteria are used for decisions.

### Q: Can PolicyContext be forged?

**A: Not from the agent side.** PolicyContext is generated by OWS itself. Agents only send requests like `ows sign --chain X --wallet Y --message Z` and do not directly interact with the JSON structure of PolicyContext.

### Q: What happens if an exception occurs during policy evaluation?

**A: It results in a Deny.** The try-catch in `evaluate.ts` catches all policy exceptions and returns `{ allow: false, reason: "Policy X threw: ..." }`. The exception details are recorded in the audit log.

### Q: Can the audit log be deleted to erase deny records?

**A: Prohibited by SQLite triggers.** DELETE/UPDATE are immediately rolled back with `RAISE(ABORT)`. If the log file itself is deleted directly, inconsistencies with hashes recorded on-chain via future Merkle Root extensions can be detected.

### Q: What happens when the 5-second timeout is exceeded?

**A: OWS automatically returns a Deny.** Even if an external API (such as a KYC service) is slow, OWS's 5-second limit will block the signing. This is OWS core behavior and does not need to be controlled on the OWS Hooks side.

### Q: What happens to an agent with an ERC-8004 reputation of 0?

**A: It depends on the `ERC8004_MIN_REPUTATION` setting.** With the default threshold of 50, a registered agent with a reputation of 0 will be blocked. Setting the threshold to 0 allows any registered agent through.

### Q: Can the preceding policy in a Policy Chain be deceived?

**A: ChainResults is an internal runtime object and is not accessible from agents.** Only the result returned by the preceding policy is passed to the next one; there is no way for an agent to intervene in this flow.

### Q: What happens if a mock policy (KYC/AML) is used in production?

**A: Mocks are for demo purposes only.** In production, replace them with external APIs. The replacement point is clear (a single line change to a fetch call), and the policy interface remains the same.

### Q: How do you add a new policy?

**A: Create a new file in `src/policies/` and add it to the policies array in `main.ts`.** Policies simply need to conform to the `{ name, evaluate(ctx, chainResults) }` interface.

### Q: What happens if OWS updates and the executable specification changes?

**A: Just update the PolicyContext/PolicyResult types in `types.ts`.** Since policy logic depends on types, TypeScript will detect all inconsistencies once the types are updated.
